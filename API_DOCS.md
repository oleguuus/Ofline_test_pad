# 📖 Спецификация API (LocalTest) — Этап 1

**Базовый URL:** `http://localhost:8000`  
**Формат обмена данными:** `application/json`

---

## 1. Тесты (Tests)

### 1.1 Получить список всех тестов
Возвращает список всех тестов в БД, за исключением удаленных (`is_deleted: true`). Вопросы внутрь списка не включаются для экономии трафика.

*   **Метод:** `GET`
*   **Эндпоинт:** `/api/tests/`
*   **Ответ (200 OK):**
```json
[
  {
    "title": "Математика: Итоговый экзамен",
    "description": "Экзамен за 1 семестр",
    "cover_image_path": null,
    "tags_json": { "tags": ["math", "exam"] },
    "time_limit_minutes": 60,
    "id": 1,
    "is_deleted": false,
    "created_at": "2026-04-22T12:00:00",
    "updated_at": "2026-04-22T12:00:00"
  }
]
```

### 1.2 Создать новый тест
Создает "оболочку" для тестирования. 

*   **Метод:** `POST`
*   **Эндпоинт:** `/api/tests/`
*   **Тело запроса (JSON):**
    *   `title` (строка, **обязательно**)
    *   `description` (строка, опционально)
    *   `cover_image_path` (строка, опционально)
    *   `tags_json` (объект/словарь, опционально)
    *   `time_limit_minutes` (число, опционально - null означает без лимита)

**Пример запроса:**
```json
{
  "title": "История Рима",
  "description": null,
  "time_limit_minutes": 45
}
```
*   **Ответ (201 Created):** Возвращает созданный тест (вместе с присвоенным `id`).

### 1.3 Получить тест с вопросами
Используется, когда преподаватель заходит в редактор конкретного теста, или когда тест нужно загрузить для студента. Возвращает сам тест и массив всех активных вопросов внутри.

*   **Метод:** `GET`
*   **Эндпоинт:** `/api/tests/{test_id}`
*   **Ответ (200 OK):**
```json
{
  "id": 1,
  "title": "Математика: Итоговый экзамен",
  "time_limit_minutes": 60,
  "created_at": "2026-04-22T12:00:00",
  "updated_at": "2026-04-22T12:00:00",
  "is_deleted": false,
  "questions": [
    {
      "id": 1,
      "test_id": 1,
      "type": "single",
      "text": "Сколько будет 2 + 2?",
      "content_payload": {
        "options": ["3", "4", "5"],
        "correct_option": "4"
      },
      "is_deleted": false,
      "is_required": true,
      "order_num": 0
    }
  ]
}
```
*   *Примечание:* Если тест удален или не существует, вернет `404 Not Found`.

### 1.4 Удалить тест (Мягкое удаление)
"Удаляет" тест для интерфейса, но физически оставляет его в БД для отчетов (переводит флаг `is_deleted` в `true`).

*   **Метод:** `DELETE`
*   **Эндпоинт:** `/api/tests/{test_id}`
*   **Ответ (204 No Content):** Никакого тела ответа, просто успешный статус-код.

## 2. Вопросы (Questions)

### 2.1 Добавить новый вопрос в тест
*   **Метод:** `POST`
*   **Эндпоинт:** `/api/tests/{test_id}/questions/`
*   **Особое внимание на поле `content_payload`**: Бэкенд не навязывает жесткую схему вариантов ответов. Фронтенд вправе присылать в это поле **любой валидный JSON-объект**. Формат содержимого зависит от типа вопроса (`type`).
*   **Тело запроса (JSON):**
    *   `type` (Enum: "single", "multiple", "number", "text", "match", "sequence") - **обязательно**
    *   `text` (строка) - текст вопроса, **обязательно**
    *   `content_payload` (объект) - структура вариантов ответов и ключей для проверки, **обязательно**
    *   `image_path` (строка, опционально)
    *   `comment` (строка, опционально)
    *   `is_required` (boolean, по умолчанию true)
    *   `order_num` (число, логический порядок отображения)

**Пример запроса (Вопрос типа `multiple`):**
```json
{
  "type": "multiple",
  "text": "Выберите языки программирования",
  "order_num": 1,
  "is_required": true,
  "content_payload": {
    "options": [
      { "id": "opt1", "text": "Python", "is_correct": true },
      { "id": "opt2", "text": "HTML", "is_correct": false },
      { "id": "opt3", "text": "Java", "is_correct": true }
    ],
    "explanation": "HTML это язык гипертекстовой разметки, а не ЯП."
  }
}
```
*   **Ответ (201 Created):** Возвращает структуру созданного вопроса с присвоенным `id` и `test_id`.

## 💡 Основные концепции для Front-End разработчика
1. **Soft Delete:** Когда вы вызываете `DELETE`, запись никуда не исчезает, мы просто меняем флаг. Это значит, что если запросить `GET /api/tests/`, этот тест больше не вернется в массиве, но старые сессии все еще смогут на него ссылаться.
2. **Гибкость типов вопросов:** Логику отображения вариантов ответов и проверки правильности (client-side) фронтенд закладывает в структуру `content_payload`. Бэкенд на данном этапе выступает надежным хранилищем этих "ответов-конфигов".
